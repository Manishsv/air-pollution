# AirOS Pilot Store Restore Design

## Purpose

This document defines a **future** restore model for the AirOS pilot file-backed store (`FileAirOsStore`).

Today, AirOS supports:

- `store backup`
- `store inspect-backup`
- `store verify-backup`
- `store restore-dry-run` (no writes)

**Actual restore is not implemented yet.** Restore is sensitive because it writes runtime state and can overwrite or merge records, runs, outputs, validation receipts, and audit events. Restore must remain **review-support only** and must not imply approval, certification, truth, or production recoverability.

## Current lifecycle

Proposed safe lifecycle:

backup  
→ inspect-backup  
→ verify-backup  
→ restore-dry-run  
→ restore (**future work**)

Briefly:

- **backup**: creates an operational zip snapshot of known JSONL store files + `store_manifest.json` + safety notes.
- **inspect-backup**: summarizes metadata and counts without extracting or writing anything.
- **verify-backup**: recomputes file hashes and checks internal consistency (not a signature).
- **restore-dry-run**: checks what *would* be restored to a target directory and detects collisions (no writes).
- **restore (future)**: writes store files and metadata according to an explicit restore mode and policy.

## Why actual restore is sensitive

Restore is sensitive because it:

- **writes runtime data** (records, outputs, runs, validation receipts, audit events)
- can **overwrite** an existing local store, causing loss of local context
- may **merge** data from another environment, introducing conflict handling and provenance risks
- may lead operators to mistake restored data as **newly generated** local runtime evidence
- may restore data that is **stale, incomplete, redacted, or from a different trust context**
- must not imply approval, certification, or that data is “true” or “authorized”

## Restore modes under consideration

### 1) Restore to empty directory only (recommended first)

Safest default:

- target directory must **not exist** or must be **empty**
- copy store files from the backup into the target directory
- write restore metadata (`restore_metadata.json`) alongside the restored JSONL files
- **no merge**, **no overwrite**

This mode reduces accidental data loss and avoids conflict policy complexity.

### 2) Restore with explicit overwrite (higher risk)

Higher risk; should never be default:

- target directory may contain existing store files
- requires an explicit `--force`
- should create an **automatic pre-restore backup** of the target before writing
- should write a `restore_report.json` (what was replaced, hashes before/after, warnings)

### 3) Restore as imported copy (safer for review)

Safer posture for review and analysis:

- writes to a new target directory (or a target subdirectory)
- explicitly marks the store as **imported/restored**, not locally generated
- writes `restore_metadata.json` stating the provenance of the restored store
- avoids any implication that data was generated by this node’s local runtime

### 4) Merge restore (defer)

Defer until conflict policy is mature:

- requires explicit conflict rules for duplicate IDs
- must preserve audit continuity without rewriting history
- requires clear semantics for “latest-write-wins” vs immutability expectations
- must not silently drop or overwrite evidence

## Required restore checks (preconditions)

Before any write operation, restore should require:

- backup zip is safe (no traversal/absolute paths/symlinks)
- backup verification passes (hashes + internal consistency)
- required files exist (`README.md`, `store_manifest.json`, `safety_notes.md`)
- no unexpected secret-like files or unknown payloads
- target directory policy passes (empty-only by default)
- operator acknowledgement that restore is **not** approval, certification, or truth

## Restore metadata

Future restore should write a small metadata file such as `restore_metadata.json` (or `restore_manifest.json`) into the target directory.

Suggested fields:

- `restored_at`
- `source_backup_id`
- `source_backup_created_at`
- optional `source_backup_sha256` (hash of the backup zip bytes) if useful
- `restore_mode` (empty-only / overwrite / imported-copy / merge)
- `files_restored` (paths + hashes + sizes)
- `warnings`
- safety note: restored runtime data is operational evidence, not approval

Identity fields (e.g., `restored_by`) should only be added when an identity & trust model exists.

## Audit implications

Current pilot store uses `audit_events.jsonl` as an application/runtime audit stream. There is no separate system audit log for store maintenance.

Restore design principles:

- **Do not rewrite** existing audit events from the backup.
- Imported/restored stores should preserve original audit events and add **restore metadata separately**.
- If the target store is an existing runtime store, a future restore may need to append a single “restore occurred” event; however this should be carefully designed to avoid implying approvals.

## Relationship to evidence bundles

- **Evidence bundles** are run/deployment-scoped review artifacts. They are portable for review and sharing (with redaction).
- **Store backups** are operational snapshots of the runtime store directory.
- Evidence bundles should **not** be restored as stores.
- **Redacted evidence bundles** must **never** be restored as runtime stores.

## Safety posture

Restoring a store must not authorize or automate:

- fund release
- penalties / recovery
- emergency orders / evacuations
- demolitions or forced actions
- blacklisting
- public disclosure without authorization
- any final government decision

Restore must not claim:

- legal attestation
- signer identity (no signing here)
- production recoverability guarantees

## Recommended implementation sequence

- **Phase 1**: restore-dry-run only (already implemented)
- **Phase 2**: restore to empty directory only
- **Phase 3**: explicit overwrite with `--force` + automatic pre-restore backup + restore report
- **Phase 4**: imported-copy mode with explicit restore metadata and “not locally generated” posture
- **Phase 5**: merge restore, only after conflict policy and persistent database work

## Open questions

- Should restore be allowed if the target store is not empty?
- Should restore require an explicit acknowledgement flag (e.g., `--acknowledge-no-approval`)?
- Should restore append a restore event into `audit_events.jsonl`, or only write separate restore metadata?
- How should duplicate run/output/receipt IDs be handled across stores?
- Should restore preserve file modification timestamps?
- Should restored stores be read-only by default?
- Who is authorized to restore in beta/production, and how is that enforced (RBAC, audit)?
- How does restore interact with future signed backups/snapshots (identity & trust integration)?
